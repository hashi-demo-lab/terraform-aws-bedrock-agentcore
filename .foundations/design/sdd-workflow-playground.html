SDD Pipeline Overview

AI-powered HCP Terraform development using Specification-Driven Development (SDD) — agents follow a structured methodology instead of generating code ad hoc, producing auditable, repeatable infrastructure.

Specification-Driven Development

Every task follows 4 sequential phases: Clarify requirements, Design the solution, Build with TDD, and Validate quality. The design document is the single source of truth that all downstream agents follow.

SDD produces a single specs/{FEATURE}/design.md containing 7 mandatory sections: Purpose & Requirements, Resources & Architecture, Interface Contract, Security Controls, Test Scenarios, Implementation Checklist, and Open Questions. This document is the sole artifact passed between the Plan and Implement phases -- no separate specification, plan, contract, or task files exist.

  • Phase 1 (Clarify): Parse requirements, create GitHub issue, launch 3-4 concurrent research agents, ask up to 4 clarification questions
  • Phase 2 (Design): Launch design agent with research findings, produce design.md, present summary for human approval
  • Phase 3 (Build): Write tests first (TDD), then implement checklist items sequentially with developer subagents
  • Phase 4 (Validate): Run full pipeline -- terraform fmt, terraform validate, terraform test, tflint, trivy, terraform-docs
## Quality Gates Between Phases
| Gate                  | Condition                                    |
| Understand -> Design  | No unresolved [NEEDS CLARIFICATION] markers  |
| Design -> Build+Test  | Design document approved, no CRITICAL issues |
| Build+Test -> Validate| terraform validate passes, all items complete|
| Validate -> Complete  | All validation checks pass                   |
No implementation begins until a human approves the design at the gate between Phase 2 and Phase 3.
Source: .foundations/memory/module-constitution.md (Section 6.3)

Orchestrator + Subagents

The orchestrator routes tasks to workflow skills and spawns specialized subagents (Researcher, Designer, Developer, Validator) for each phase. Subagents run in isolated contexts with scoped tools.

The orchestrator skill (e.g. tf-module-plan) never writes code itself -- it dispatches to specialized subagents defined as markdown files in .claude/agents/. Each agent declares its own model, color, allowed skills, and scoped tool list in YAML frontmatter. The orchestrator passes context via $ARGUMENTS and verifies outputs via Glob/Grep, never reading file contents directly.

  • tf-module-research: Answers ONE research question per instance using MCP tools (provider docs, AWS docs, registry). Returns findings in-memory.
  • tf-module-design: Reads constitution + template, produces the 7-section design.md. Uses model: opus.
  • tf-module-developer: Executes one checklist item from design.md Section 6. Runs fmt, validate, test after each item.
  • tf-module-validator: Runs full validation pipeline, quality scoring across 6 dimensions, auto-fixes unambiguous issues.

Layered Guardrails

Every agent operates within guardrails: Constitution files define rules, MCP Tools control data access, Design Templates enforce structure, and Human Approval Gates provide checkpoints.

The Module Constitution (.foundations/memory/module-constitution.md) is an 8-section governance document covering Core Principles, Code Standards, Security & Compliance, Version Management, Testing, Change Management, Operational Standards, and Governance. It is loaded as P1 context (always present) and enforces non-negotiable rules like "no provider blocks in root modules" and "encryption enabled by default".

  • Constitution: Security-first defaults (Section 3.2), file organization (Section 2.1), variable conventions (Section 2.3), tagging (Section 3.3)
  • MCP Tools: Terraform registry search/details and AWS documentation search/read -- agents cannot access arbitrary APIs
  • Design Templates: 7-section template with strict rules -- variable names appear exactly once, no cross-section line references
  • Human Gates: Design approval between Phase 2-3, PR review between Phase 4 and merge

Concrete Outputs

The system produces real Terraform artifacts: *.tf files, design.md documents, test files, validation reports, and pull requests -- all from natural language requirements.

The implementation phase follows standard HashiCorp module structure with raw resources, secure defaults, and conditional creation patterns. The validator agent scores the output across 6 weighted dimensions before creating a PR, ensuring every artifact meets production quality standards.

  • Module files: main.tf, variables.tf, outputs.tf, locals.tf, versions.tf, data.tf
  • Test files: tests/basic.tftest.hcl, tests/complete.tftest.hcl, tests/edge_cases.tftest.hcl, tests/validation.tftest.hcl

Skills: Domain Knowledge for Agents

SKILL.MD files package reusable expertise that the agent selects by matching name and description — giving agents domain-specific knowledge without bloating their base context window.

Name & Description Matching

Agents discover skills by scanning name and description fields. The orchestrator matches the current task context against available skills and loads only those that are relevant.

concept-skills

Every SKILL.MD file begins with YAML frontmatter containing name, description, and optionally user-invocable and argument-hint fields. The agent runtime indexes these fields and presents them as available capabilities. When a task matches, the full skill body (Instructions, Output, Constraints, Examples, Context) is loaded into the agent's context.

Progressive Disclosure

Metadata is loaded first (name, description, triggers). Full skill content is only loaded on-demand when the agent decides to invoke the skill, preserving context window space.

Skills follow a two-phase loading pattern. Phase one exposes only YAML frontmatter (name + description, roughly 50-100 tokens). Phase two loads the full skill body only when the agent decides to invoke it. This prevents context window bloat -- a workspace with 12+ skills would consume thousands of tokens if all were pre-loaded.

  • Phase 1 (index): Agent sees name, description, and argument-hint -- enough to decide relevance
  • Phase 2 (load): Full Instructions, Workflow, Output, Constraints, Examples, and Context sections are injected
  • Subagents load skills themselves (P9 principle) -- the orchestrator never pre-loads skill content for them

Subagents: Custom Specialists

Markdown files that define specialized AI assistants with scoped tools, models, and permissions — each agent gets only the capabilities it needs, enforcing least-privilege by design.

Isolated Execution

Each subagent runs in its own context window with a focused task, limited tools, and returns results to disk. Key benefit: separation of duty and context extension through task breakdown.

concept-subagents

Subagents are spawned by the orchestrator and run in completely isolated context windows. The research agent uses only ~23% of context capacity, the design agent ~35%, the developer ~31%, and the validator ~22%. This isolation means each agent gets a fresh, focused context rather than competing for space in an overloaded single window.

  • Research agents run concurrently (3-4 at once), each answering exactly ONE question -- findings returned in-memory, never written to disk
  • Developer agents run sequentially per checklist item, each reading design.md fresh and producing .tf file changes
  • The orchestrator verifies subagent output via Glob (file exists?) and Grep (sections present?) -- it never reads the full content
  • If a subagent fails, the orchestrator can re-launch it once without crashing the entire workflow
Source: .foundations/design/tf-plan-module-flow.md

Scoped Tools

Subagents declare their allowed tools via allowedTools in frontmatter. The orchestrator never grants more access than the subagent needs for its specific task.

Each agent's tools: frontmatter field is an explicit allowlist. The research agent gets MCP search tools but no Write/Edit. The design agent gets file tools plus MCP but no destructive operations. This enforces least-privilege at the agent level -- a research agent physically cannot modify source files.

  • tf-module-research: Read, Bash, Grep, Glob + 14 MCP tools (terraform search/get modules, providers, private registry, policies + AWS search/read/recommend/availability)
  • tf-module-design: Read, Write, Edit, Bash, Glob, Grep + 6 MCP tools (terraform + AWS docs)
  • tf-module-developer: Read, Write, Edit, Bash, Glob, Grep + 6 MCP tools (module/provider search, AWS docs)
  • tf-module-validator: Read, Write, Edit, Bash, Glob, Grep -- no MCP tools (works entirely with local files)

Context Engineering

Controlling what information enters an agent's context window — focused context produces higher-quality output and avoids the hallucination risks of noisy, oversized prompts.

Priority System

Context is loaded by priority: P1 (constitutions, always loaded), P2 (skills, loaded on demand), P9 (subagent reads skills themselves). This prevents context window overflow.

concept-context-engineering

Without priority-based loading, a single agent trying to hold the constitution, all 12+ skills, all research findings, and the full conversation history would hit 92% context capacity, leaving almost no room for reasoning. The priority system ensures each agent only loads what it needs for its specific task.

  • P1 (Always loaded): Constitution rules from .foundations/memory/module-constitution.md -- non-negotiable governance
  • P2 (On-demand): Skills loaded when the agent matches task to skill name/description
  • P9 (Self-service): Subagents read their own skills via the skills: frontmatter declaration -- the orchestrator never pre-loads context for them
  • Research findings are passed in-memory (not written to disk) to avoid terminology drift from intermediate files
Source: .foundations/design/tf-plan-module-flow.md

Single vs Multi-Agent

Single-agent patterns load everything into one context. Multi-agent patterns distribute context across specialized subagents, each with only the knowledge they need.

The single-agent approach loads constitution + all skills + task history + conversation into one window, hitting ~92% capacity. The multi-agent approach splits this: the orchestrator holds routing logic (~15% ctx), research agents each hold one question + MCP results (~23% ctx), the design agent holds constitution + template + findings (~35% ctx). Total information processed is actually greater, but each agent's context stays focused.

  • Single agent: Everything competes for limited context -- constitution, 12 skills, all history. Quality degrades as the window fills.
  • Multi-agent: Each subagent gets a fresh context with only its required knowledge. Research at 23%, Design at 35%, Developer at 31%, Validator at 22%.
  • The orchestrator directs but doesn't accumulate (P6) -- it passes file paths and short summaries, never reads full content

Concurrency Patterns

Fan-out/fan-in, pipeline, and map-reduce patterns for agent orchestration — composable patterns let agents divide complex infrastructure tasks into parallel, independently verifiable subtasks.

Concurrent Agent Patterns

Fan-out/fan-in (multiple research agents), pipeline (sequential handoffs), and map-reduce (parallel work + merge). The orchestrator controls concurrency -- subagents never spawn their own children.

concept-concurrency

The tf-plan-module skill demonstrates fan-out/fan-in concretely: Step 7 launches 3-4 tf-module-research agents concurrently, each investigating one question (provider docs, AWS best practices, registry patterns, edge cases). All agents run in foreground (they need MCP tools), return findings in-memory, and the orchestrator collects all results before launching the single design agent.

  • Fan-out/fan-in: 3-4 research agents dispatched in parallel, each answering ONE question. Results collected before next phase.
  • Pipeline: Research -> Design -> Test-Writer -> Developer(s) -> Validator. Each stage's output feeds the next.
  • Map-reduce: Developer agents can run concurrently on independent checklist items, with the orchestrator running terraform test after each.
  • Orchestrator verifies via Glob after each fan-in -- "do the expected output files exist?"

Throughput Optimization

Maximizing agent productivity with model selection and verification patterns — matching model capability to task complexity controls cost while maintaining output quality.

Keeping Agents Productive

Use concurrent research (3-4 agents), checkpoint commits after each item, trust-but-verify gates (orchestrator validates subagent output).

concept-throughput

Throughput is maximized through three mechanisms: concurrent agent execution, model tiering to match cost to complexity, and checkpoint commits that create safe rollback points. The developer agent runs terraform test internally (its own feedback), and then the orchestrator also runs terraform validate + terraform test as a trust-but-verify gate.

  • Concurrent execution: Research agents run in parallel. Developer agents can run concurrently on independent checklist items.

  • Trust-but-verify: Developer runs fmt/validate/test internally; orchestrator re-runs them as authoritative check

Decision Framework

When to use skills vs subagents vs MCP tools — choosing the right extension mechanism prevents context pollution and keeps each agent focused on its responsibility.

Skills vs Subagents vs MCP

Skills for domain knowledge injection. Subagents for parallel isolated work. MCP for external API access. Key rule: subagents read skills themselves (P9) -- don't pre-load context for them.

concept-decision-framework

The decision matrix maps task characteristics to the right execution model. Interactive + focused tasks stay in the main conversation. Broad tasks with parallel subtasks use subagents. Reusable domain patterns use skills. Fully autonomous CI/CD uses headless mode. MCP tools are the bridge to external systems (Terraform registry, AWS docs) and are always scoped per-agent.

  • Skills: Inject domain knowledge into an agent's context. Examples: tf-implementation-patterns (VPC, IAM, S3 patterns), terraform-style-guide, tf-security-baselines
  • Subagents: Isolated context windows for parallel work. Each defined in .claude/agents/*.md with model, tools, and skills declarations.
  • MCP tools: External API access via .mcp.json. Two servers configured: terraform (Docker, hashicorp/terraform-mcp-server:0.4.0, --toolsets=all) and aws-knowledge-mcp-server (HTTP, knowledge-mcp.global.api.aws)
  • Headless mode: Fully autonomous pipelines for CI/CD. No shared state, true parallelism.

The Agentic Loop

Plan, Execute, Observe, Reflect, Repeat — the core reasoning loop that enables agents to self-correct and converge on working infrastructure through iterative validation.

Outer and Inner Loops

The orchestrator drives the outer loop while subagents handle inner loops. Human approval gates interrupt the loop at design review and PR creation.

concept-agentic-loop

The outer loop is the orchestrator's 4-phase pipeline: Plan, Execute (dispatch subagents), Observe (verify outputs), Reflect (check quality gates), Repeat (fix or advance). The inner loop is the developer subagent's TDD cycle: write code, run terraform fmt, run terraform validate, run terraform test, check results, fix if needed. Human gates interrupt at two points.

  • Outer loop (orchestrator): Dispatches subagents, verifies via Glob/Grep, runs authoritative terraform test, advances to next phase or retries
  • Inner loop (developer): Reads design.md, implements checklist item, runs fmt/validate/test, marks [x] in Section 6, reports pass/fail counts
  • Human Gate 1: Design Review -- user approves design.md before any implementation begins (Phase 2 -> Phase 3)
  • Human Gate 2: PR Creation -- validator scores >= 7.0, orchestrator creates PR for human code review

MCP tools, deny permissions, dev containers, git integration, and agent identity boundaries

Autonomous agents need hard boundaries. Without guardrails, an AI agent with write access to your infrastructure is indistinguishable from an insider threat. This chapter layers eight complementary controls — from network isolation and secret detection to policy-as-code and identity federation — so agents operate with the minimum privilege needed and every action is auditable.

Deny Permissions

Restricting what agents can do — deny by default, allow explicitly, so agents cannot execute destructive operations even if hallucinated instructions suggest otherwise.

Immutable Guardrails

Agents are denied write access to verification and configuration files. If an agent could modify them, it could effectively remove its own constraints.

concept-deny-permissions

The settings.local.json deny list uses three permission verbs per file — Write, Update, and Edit — to close every possible mutation path. This triple-deny pattern ensures that regardless of how the agent attempts to modify a file, the operation is blocked.

  • Each protected file has three deny entries: Write(.gitignore), Update(.gitignore), Edit(.gitignore)
  • Four files protected: .gitignore, .tflint.hcl, .pre-commit-config.yaml, .mcp.json
  • Two MCP operations denied: mcp__terraform__create_workspace_variable, mcp__terraform__create_run
  • Deny rules take precedence over allow rules — even if an allow pattern would match, the deny blocks it
"deny": [
  "Write(.gitignore)",
  "Update(.gitignore)",
  "Edit(.gitignore)",
  "Write(.tflint.hcl)",
  "Update(.tflint.hcl)",
  "Edit(.tflint.hcl)",
  "Write(.pre-commit-config.yaml)",
  ...
]
Source: .claude/settings.local.json

Self-Modification Blocked

Agents cannot weaken linting rules, disable pre-commit hooks, or alter MCP configurations. This prevents gradual erosion of quality gates during long-running tasks.

Without deny permissions, a sufficiently capable agent could reason that removing a failing linting rule is the "faster" path to completion. The deny list prevents this self-modification vector. The agent also has disableAllHooks: true set to prevent hooks from being used as an attack surface, while pre-commit hooks are installed and enforced through the explicit allow of pre-commit run.

  • .tflint.hcl protected: agent cannot weaken linting rules to make code pass
  • .pre-commit-config.yaml protected: hook configuration is immutable
  • .mcp.json protected: agent cannot grant itself additional MCP tool access
  • create_workspace_variable denied: agent cannot inject environment variables into HCP Terraform workspaces

Git Integration

Checkpoint Commits

After each phase completion and checklist item, the agent creates a checkpoint commit providing fine-grained rollback points.

checkpoint-commit.sh

The checkpoint-commit script creates deterministic commit messages following the pattern {prefix}({feature}): complete {step} artifacts. It uses --no-verify to skip pre-commit hooks on intermediate saves — these are partial work in progress, and Phase 4 runs pre-commit run --all-files as the explicit quality gate.

Agent Identity & Workload Federation

Credential-free agent operations through OIDC federation — agents never see long-lived secrets, receiving short-lived tokens scoped to a single run.

Agent Never Sees Credentials

The agent operates at the HCP Terraform API layer. It never handles AWS keys, Azure secrets, or GCP service accounts directly.

concept-agent-identity

The module constitution explicitly forbids credential handling: "Modules MUST NOT require consumers to pass credentials" and "MUST NOT generate static credential variables." The agent interacts with HCP Terraform via the TFE_TOKEN API token, and cloud provider credentials are injected by the platform at run time via dynamic provider credentials.

  • Constitution rule: "MUST NOT generate static credential variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY)"
  • Modules inherit providers from consumers — never include provider {} blocks in root modules
  • Consumer constitution: "Provider authentication MUST use HCP Terraform dynamic provider credentials"
  • Variables accepting secrets MUST be marked sensitive = true
Source: .foundations/memory/module-constitution.md, Section 3.1

OIDC Federation

Workspace-to-provider trust via OpenID Connect. Each workspace has a federated identity granting just enough access to the target cloud provider.

HCP Terraform workspaces use OIDC federation to obtain cloud provider credentials without storing long-lived secrets. The consumer constitution mandates: "Dynamic provider credentials MUST be used — never static AWS credentials." This means the agent never has access to persistent cloud credentials, even at the workspace level.

  • Consumer constitution: "MUST NOT define static credential variables"
  • Cross-account access MUST use assume_role in provider configuration
  • Dynamic provider credentials configured at the workspace level in HCP Terraform
  • Secrets managed via HCP Terraform variable sets or workspace variables marked sensitive
Source: .foundations/memory/consumer-constitution.md, Section 3.1

Dynamic Credentials Per Run

Every HCP Terraform run receives short-lived, dynamically generated credentials. Tokens expire after the run completes.

Each HCP Terraform run receives time-bounded credentials through the OIDC trust relationship. These credentials exist only for the duration of the plan or apply operation and are automatically revoked when the run completes. The create_workspace_variable MCP operation is explicitly denied so the agent cannot tamper with credential configuration.

  • Credentials are generated per-run — no persistent secrets in workspace configuration
  • mcp__terraform__create_workspace_variable is in the deny list to prevent credential injection
  • Consumer code must use cloud {} block for HCP Terraform backend
  • State locking handled by HCP Terraform — no additional configuration needed

Private Module Registry

Curated, versioned modules as the only source for consumer compositions — agents compose from approved building blocks rather than inventing bespoke resource configurations.

Registry as Guardrail

The private registry is not just a convenience — it's a security boundary. Consumers compose from pre-approved, security-scanned modules. Raw resource blocks are prohibited by constitution.

The consumer constitution explicitly prohibits raw resource blocks: "ALL infrastructure MUST be provisioned via private registry modules: source = \"app.terraform.io/<org>/<name>/<provider>\"." Only glue resources (random_id, random_string, null_resource, terraform_data, time_sleep) are exempted. This means agents composing infrastructure cannot invent bespoke resources.

  • Raw resource blocks are PROHIBITED except for glue resources
  • Module sources MUST reference the private registry: app.terraform.io/<org>/<name>/<provider>
  • Module selection MUST be justified by research findings — never assumed
  • Consumers do NOT write .tftest.hcl files — validation is via terraform validate and sandbox deployment
Source: .foundations/memory/consumer-constitution.md, Section 1.1

Version Governance

Semantic versioning with ~> X.Y pins. Breaking changes require major version bumps. CHANGELOG review before upgrades. Sentinel policies enforce registry-only module sources.

Version constraints differ between module authors and consumers by design. Module authors use >= (minimum version) to maximize compatibility. Consumers use ~> (pessimistic constraint) for stability. Major version upgrades require a design document update and review.

  • Modules: version = ">= 5.0" in required_providers (maximum consumer compatibility)
  • Consumers: version = "~> X.Y" for modules (pessimistic constraint, stability)
  • CHANGELOG of consumed modules SHOULD be reviewed before version bumps
  • Consumer must not override module secure defaults: "MUST NOT set encryption_enabled = false"

Sentinel Policy as Code

Automated policy enforcement on every HCP Terraform run — agent-generated plans are evaluated against organisational rules before any infrastructure change is applied.

Policy Levels

Hard-mandatory: blocks apply, no override. Soft-mandatory: blocks apply, manager can override. Advisory: logs warning, does not block. All evaluated between plan and apply.

Sentinel policies are evaluated after terraform plan and before terraform apply on every HCP Terraform run. Hard-mandatory policies cannot be overridden by anyone — they enforce absolute security baselines like encryption requirements. Soft-mandatory policies allow a manager to override with justification, supporting legitimate exceptions. Advisory policies provide visibility without blocking.

  • Hard-mandatory: encryption required on all storage, no public S3 buckets, no unencrypted EBS
  • Soft-mandatory: registry-only module sources — override requires manager approval
  • Advisory: cost estimation thresholds — warn on monthly cost exceeding budget
  • All policy types evaluated on every run — no way for agents to skip the check
Source: .foundations/memory/consumer-constitution.md, Section 7.3

Agent-Proof Enforcement

Sentinel runs at the HCP Terraform platform layer. Agents interact via API and cannot circumvent policy checks. This is defense-in-depth beyond constitution rules.

Constitution rules are enforced at the agent prompt level and can theoretically be bypassed through prompt injection. Sentinel provides a second, independent enforcement layer at the platform level. Even if an agent were to ignore its constitution, the plan would still be blocked by hard-mandatory Sentinel policies before any infrastructure change is applied.

  • Defense-in-depth: constitution (agent layer) + Sentinel (platform layer) + deny permissions (tool layer)
  • mcp__terraform__create_run is denied — agents cannot even trigger runs directly
  • Run tasks can validate additional checks before plan/apply proceeds
  • Sentinel policies MAY enforce organizational compliance at the workspace level

Vault Radar & Pre-Commit

Secret detection before code leaves the developer's machine — agents are blocked from committing leaked credentials, catching mistakes before they reach version control.

Vault Radar

HashiCorp Vault Radar scans staged files for 100+ secret patterns: AWS keys, API tokens, private keys, database credentials. Runs as a pre-commit hook — secrets never reach the repository.

Vault Radar is configured as a local pre-commit hook that runs vault-radar scan git pre-commit on every commit. It only activates when the VAULT_RADAR_LICENSE environment variable is set (passed from the host via devcontainer remoteEnv). If a secret is detected, the commit is blocked with a non-zero exit code.

  • Hook type: local (not from a remote repo), always_run: true
  • Conditional execution: skipped if VAULT_RADAR_LICENSE is not set (graceful degradation)
  • License env var passed from host: "VAULT_RADAR_LICENSE": "${localEnv:VAULT_RADAR_LICENSE:}"
  • Catches: AWS keys, API tokens, passwords, private keys, database credentials (100+ patterns)
- id: vault-radar-scan
  name: Vault Radar scan
  entry: bash -c 'if [ -n "$VAULT_RADAR_LICENSE" ]; then
    vault-radar scan git pre-commit "$@" || { exit "$?"; };
  else
    echo "Vault Radar scan skipped (VAULT_RADAR_LICENSE not set)";
  fi'
  language: system
  always_run: true

Spec-Driven Development

The 4-phase methodology where the design document is the single source of truth

4-Phase Pipeline

Clarify, Design, Build, Validate — with human approval gating implementation. Agents follow a phased pipeline so ambiguity is resolved before any code is generated.

Every task follows the same 4-phase methodology. Phase 1 (Clarify) gathers requirements. Phase 2 (Design) produces a design doc. Phase 3 (Build) implements via TDD. Phase 4 (Validate) runs quality checks. The design doc is the single source of truth.

concept-sdd-overview

Two orchestrator skills split the pipeline: /tf-{wf}-plan covers Phases 1-2 (Clarify + Design), while /tf-{wf}-implement covers Phases 3-4 (Build+Test + Validate). The ONLY artifact passed between them is specs/{FEATURE}/design.md -- no shared state, no intermediate research files.

  • Phase 1 launches 3-4 concurrent research agents that return findings in-memory (never written to disk)
  • Phase 2 produces a single design.md with 7 mandatory sections, gated by human approval
  • Phase 3 follows TDD: test-writer agent runs before developer agent, creating a RED baseline
  • Phase 4 runs 6 validation tools (fmt, validate, test, tflint, trivy, terraform-docs) and scores quality on a 1-10 scale
design.md ──► tf-{wf}-implement orchestrator
  (Sections 2, 3, 5) ──► tf-module-test-writer
  (Sections 2, 3, 4, 6) ──► tf-module-developer (per checklist item)

HCP Terraform Setup

Configure HCP Terraform with an isolated testing environment — agents need a sandbox workspace where plan/apply cycles run without affecting production.

1. Create a Dedicated Project

Navigate to Projects in HCP Terraform and create a new project (e.g., sandbox).

A dedicated project isolates agent-created workspaces from production. The module constitution (Section 7.3) requires that sandbox workspaces follow the naming pattern sandbox_<module>_<example> and that ephemeral workspaces are deleted after testing.

  • Agents create workspaces per test cycle — a dedicated project keeps them organized and easy to clean up
  • Project-level variable sets inherit to all workspaces, avoiding per-workspace credential setup
  • The create_workspace MCP tool targets this project specifically via TFE_TOKEN scoping
Source: .foundations/memory/module-constitution.md (Section 7.3)

2. Create a Dedicated Team

Go to Settings > Teams. Create a team and assign it to your project with permissions:

Read, Create Workspaces, Delete Workspaces,
Read Variables, Read State, Write State,
Lock/Unlock Workspaces

Team permissions are the authorization boundary for the agent. The token generated for this team determines exactly what the agent can do in HCP Terraform. Note that create_workspace_variable is explicitly denied in settings.local.json — agents cannot inject variables into workspaces even if the team token would technically allow it.

  • Create/Delete Workspaces: agents create sandbox workspaces for plan/apply validation, then clean up
  • Read Variables & State: agents need to inspect workspace state during validation
  • Lock/Unlock: prevents concurrent plan/apply collisions in sandbox workspaces
  • Do NOT grant Manage Policies or Manage Policy Overrides — Sentinel enforcement must remain immutable
Source: .claude/settings.local.json (deny list)

3. Generate Team API Token

In Settings > Teams > [Your Team], click "Create a team token". Save as TEAM_TFE_TOKEN.

4. Configure Credential Inheritance

Create a Variable Set with AWS credentials and attach it to your dedicated project.

Variable sets provide credential inheritance at the project level, so every workspace the agent creates automatically receives AWS credentials. The consumer constitution mandates dynamic provider credentials and prohibits static credential variables in Terraform code.

  • Attach the variable set to the project (not individual workspaces) — new agent-created workspaces inherit credentials automatically
  • Mark all credential variables as Sensitive in HCP Terraform
  • Constitution rule: "MUST NOT define static credential variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, or equivalent)"
  • The MCP deny list blocks create_workspace_variable — agents cannot create credentials via API
Source: .foundations/memory/consumer-constitution.md (Section 3.1)

5. AWS Credentials

Prefer Dynamic Provider Credentials (OIDC federation) or a Variable Set with:

AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
AWS_REGION

Dynamic provider credentials (OIDC federation) eliminate long-lived secrets entirely. HCP Terraform uses an OIDC provider to obtain short-lived credentials that expire after each run. If OIDC is not yet configured, static credentials in a variable set work but should be rotated regularly.

  • Preferred: Configure an OIDC trust between HCP Terraform and your OIDC provider — zero static secrets
  • Fallback: Variable set with AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN marked sensitive
  • The devcontainer also passes local AWS credentials for CLI operations: ${localEnv:AWS_ACCESS_KEY_ID}, ${localEnv:AWS_SESSION_TOKEN}, ${localEnv:AWS_SESSION_EXPIRATION}
  • Cross-account access must use assume_role in provider configuration per constitution Section 3.1
