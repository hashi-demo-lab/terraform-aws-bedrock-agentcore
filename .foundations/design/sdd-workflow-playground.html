Platform engineering teams managing HCP Terraform at scale. Module authors, provider developers, and consumer teams who need repeatable, auditable infrastructure delivery.

Chapter One

Foundations

Agent architecture, skills, subagents, context engineering, and the core concepts that power autonomous Terraform development

SDD Pipeline Overview

AI-powered Terraform development using Specification-Driven Development (SDD)

Specification-Driven Development

Every task follows 4 sequential phases: Clarify requirements, Design the solution, Build with TDD, and Validate quality. The design document is the single source of truth that all downstream agents follow.

No implementation begins until a human approves the design at the gate between Phase 2 and Phase 3.

Orchestrator + Subagents

The orchestrator routes tasks to workflow skills and spawns specialized subagents (Researcher, Designer, Developer, Validator) for each phase. Subagents run in isolated contexts with scoped tools.

Blast radius containment: a subagent failure never crashes the parent workflow.

Layered Guardrails

Every agent operates within guardrails: Constitution files define rules, MCP Tools control data access, Design Templates enforce structure, and Human Approval Gates provide checkpoints.

Guardrails wrap the entire SDD engine -- no agent operates outside their boundaries.

Concrete Outputs

The system produces real Terraform artifacts: *.tf files, design.md documents, test files, validation reports, and pull requests -- all from natural language requirements.

Quality-gated: outputs must score >= 7.0 on a 10-point rubric before the PR is created.

Skills: Domain Knowledge for Agents

SKILL.MD files package reusable expertise that the agent selects by matching name and description

Name & Description Matching

Agents discover skills by scanning name and description fields. The orchestrator matches the current task context against available skills and loads only those that are relevant.

concept-skills
Skills are not pre-loaded -- the agent decides which skills to pull in based on the task at hand.

Progressive Disclosure

Metadata is loaded first (name, description, triggers). Full skill content is only loaded on-demand when the agent decides to invoke the skill, preserving context window space.

This two-phase loading keeps the agent's context lean until a skill is actually needed.

Version-Controlled

Skills live in .claude/skills/ and are version-controlled alongside the codebase. Changes to skills are reviewed via standard PR workflows.

tf-module-plan
terraform-style-guide
tf-research
Stored as markdown files under .claude/skills/ -- portable, diffable, and composable.

User-Invocable Slash Commands

Skills can be triggered directly by users via slash commands (e.g. /tf-module-plan). This gives humans the same entry points that agents use programmatically.

Slash commands bridge the gap between interactive use and autonomous execution.

Subagents: Custom Specialists

Markdown files that define specialized AI assistants with scoped tools, models, and permissions

Isolated Execution

Each subagent runs in its own context window with a focused task, limited tools, and returns results in-memory. Key benefit: blast radius containment.

concept-subagents

Scoped Tools

Subagents declare their allowed tools via allowedTools in frontmatter. The orchestrator never grants more access than the subagent needs for its specific task.

Model Selection

Different subagents can use different models based on task complexity. Research agents may use faster models while design agents use more capable ones.

In-Memory Results

Subagents return findings to the orchestrator in-memory (under 500 tokens). No intermediate files needed for simple handoffs between phases.

Context Engineering

Controlling what information enters an agent's context window

Priority System

Context is loaded by priority: P1 (constitutions, always loaded), P2 (skills, loaded on demand), P9 (subagent reads skills themselves). This prevents context window overflow.

concept-context-engineering

Single vs Multi-Agent

Single-agent patterns load everything into one context. Multi-agent patterns distribute context across specialized subagents, each with only the knowledge they need.

Concurrency Patterns

Fan-out/fan-in, pipeline, and map-reduce patterns for agent orchestration

Concurrent Agent Patterns

Fan-out/fan-in (multiple research agents), pipeline (sequential handoffs), and map-reduce (parallel work + merge). The orchestrator controls concurrency -- subagents never spawn their own children.

concept-concurrency

Throughput Optimization

Maximizing agent productivity with model selection and verification patterns

Keeping Agents Productive

Use concurrent research (3-4 agents), checkpoint commits after each item, trust-but-verify gates (orchestrator validates subagent output), and in-memory result passing (<500 tokens).

concept-throughput

Decision Framework

When to use skills vs subagents vs MCP tools

Skills vs Subagents vs MCP

Skills for domain knowledge injection. Subagents for parallel isolated work. MCP for external API access. Key rule: subagents read skills themselves (P9) -- don't pre-load context for them.

concept-decision-framework

The Agentic Loop

Plan, Execute, Observe, Reflect, Repeat

Outer and Inner Loops

The orchestrator drives the outer loop while subagents handle inner loops. Human approval gates interrupt the loop at design review and PR creation.

concept-agentic-loop

Observability

Tracking agent progress, quality, and rollback points

Progress Dashboard

GitHub issue as progress dashboard, post-issue-progress.sh updates at every step, checkpoint commits for rollback points, structured validation reports for quality tracking.

concept-observability
Chapter Two

Guardrails

MCP tools, deny permissions, dev containers, git integration, and agent identity boundaries

MCP Tools

Model Context Protocol tools for controlled external API access

Terraform MCP Server

8 tools: search/read modules, search/read providers, search policies, list organizations, read workspaces, create workspaces. Runs as Docker container inside the devcontainer.

terraform-mcp

AWS Knowledge MCP

4 tools: search_documentation, read_documentation, recommend, get_regional_availability. Provides authoritative AWS service documentation to research agents.

aws-knowledge-mcp

Allow-Listed Per Agent

MCP tools are not globally available. Each agent's configuration specifies exactly which MCP tools it can access. Research agents get documentation tools; design agents get registry tools.

concept-mcp-tools

Deny Permissions

Restricting what agents can do -- deny by default, allow explicitly

Immutable Guardrails

Agents are denied write access to verification and configuration files. If an agent could modify them, it could effectively remove its own constraints.

concept-deny-permissions

Self-Modification Blocked

Agents cannot weaken linting rules, disable pre-commit hooks, or alter MCP configurations. This prevents gradual erosion of quality gates during long-running tasks.

DENY List

Agents cannot write to these files:

  • .gitignore
  • .tflint.hcl
  • .pre-commit-config.yaml
  • .mcp.json
  • create_workspace_variable
  • create_run

ALLOW List

Explicitly permitted commands:

  • terraform init / validate / fmt / plan
  • git add / commit
  • pre-commit run
  • gh issue view
  • validate-env.sh
  • create-feature.sh

Development Containers

Reproducible, isolated environments with pre-installed toolchains

Docker-in-Docker

MCP servers run as containers within the dev container. Each MCP tool gets its own isolated runtime, preventing cross-contamination between tools.

concept-dev-containers

Consistent Toolchain

All tools pre-installed at pinned versions: terraform, tflint, trivy, Claude Code, vault-radar, and pre-commit. No version drift between developers or CI.

Env Vars for Agent Config

TFE_TOKEN, AWS_REGION, and feature flags set in the container definition, ensuring consistent agent configuration.

Team-Portable

The dev container definition is checked into the repository. Any team member or CI runner can spin up an identical environment in seconds.

Git Integration

Structured version control patterns for traceable, reversible agent work

Feature Branches per Task

Every agent task starts with a dedicated feature branch (feat/{FEATURE}), isolating agent work from main.

concept-git-integration

Checkpoint Commits

After each phase completion and checklist item, the agent creates a checkpoint commit providing fine-grained rollback points.

checkpoint-commit.sh

GitHub Issues for Tracking

Each task gets a GitHub issue as a progress dashboard. The agent posts structured updates at every step.

post-issue-progress.sh

PRs as Final Deliverable

The agent's final output is always a pull request linked to the tracking issue, subject to standard team review.

gh pr create

Agent Identity & Workload Federation

Credential-free agent operations through OIDC federation

Agent Never Sees Credentials

The agent operates at the HCP Terraform API layer. It never handles AWS keys, Azure secrets, or GCP service accounts directly.

concept-agent-identity

OIDC Federation

Workspace-to-provider trust via OpenID Connect. Each workspace has a federated identity granting just enough access to the target cloud provider.

Dynamic Credentials Per Run

Every Terraform run receives short-lived, dynamically generated credentials. Tokens expire after the run completes.

Blast Radius Contained

Workspaces are scoped to projects. An agent working on module A cannot affect the state or resources of module B.

Constitutions Reference

Non-negotiable rules for all code generation

Module Constitution

  • File structure: main.tf, variables.tf, outputs.tf, versions.tf
  • Naming: snake_case, this for single instances
  • Security defaults: encryption enabled, public access blocked
  • tags variable required, no provider blocks, max 500 lines
module-constitution

Provider Constitution

  • Go conventions: camelCase locals, PascalCase exports
  • Plugin Framework: types.String, not raw Go types
  • Tests: basic, disappears, fullFeatures, update, validation required
provider-constitution

Consumer Constitution

  • Module-first: raw resources PROHIBITED except glue
  • Dynamic credentials, never weaken module defaults
  • default_tags required: ManagedBy, Environment, Project, Owner
consumer-constitution

Shared Principles

  • Security-first, deny-by-default
  • Single design doc as source of truth
  • Quality gates at every phase
  • Direct main/master commits prohibited

Authoring Guides

How to create new skills, agents, and constitutions

Writing a Skill

  • Create .claude/skills/[name]/SKILL.md
  • YAML frontmatter: name, description, user-invocable flag
  • Markdown body with domain knowledge and examples
  • Reference via slash command: /[skill-name]

Defining an Agent

  • Create .claude/agents/[name].md
  • YAML: name, model, allowedTools, disallowedTools
  • Markdown body = system prompt
  • Scope tools to minimum needed

Maintaining Constitutions

  • Located in .foundations/memory/
  • Owned by platform team, changes require PR review
  • Must cover: file structure, naming, security, testing
  • Always loaded (P1 priority), cannot be overridden

Design Template Structure

  • Located in .foundations/templates/
  • 6-7 mandatory sections per workflow type
  • Machine-readable format for downstream agents
  • Shared sections: Security Controls, Implementation Checklist

Prerequisites

Environment setup, HCP Terraform configuration, and your first workflow

Required Software

Software and environment variables needed to get started

Docker Desktop

Required for running the devcontainer. Download from docker.com and ensure it is running.

VS Code

Recommended IDE with devcontainer support. Install the Dev Containers extension.

Environment Variables

Two tokens required:

GITHUB_TOKEN    # GitHub PAT with repo permissions
TEAM_TFE_TOKEN  # HCP Terraform Team Token (must be a Team API Token)

Important: TEAM_TFE_TOKEN

Must be a Team API Token -- not a user token or organization token. Create at:

HCP Terraform > Settings > Teams > [Your Team] > Team API Token

HCP Terraform Setup

Configure HCP Terraform with an isolated testing environment

1. Create a Dedicated Project

Navigate to Projects in HCP Terraform and create a new project (e.g., sandbox).

2. Create a Dedicated Team

Go to Settings > Teams. Create a team and assign it to your project with permissions:

Read, Create Workspaces, Delete Workspaces,
Read Variables, Read State, Write State,
Lock/Unlock Workspaces

3. Generate Team API Token

In Settings > Teams > [Your Team], click "Create a team token". Save as TEAM_TFE_TOKEN.

4. Configure Credential Inheritance

Create a Variable Set with AWS credentials and attach it to your dedicated project.

5. AWS Credentials

Prefer Dynamic Provider Credentials (OIDC federation) or a Variable Set with:

AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
AWS_REGION

6. Shell Configuration

Add tokens to your shell profile:

# Add to ~/.bashrc or ~/.zshrc
export GITHUB_TOKEN="ghp_your_token_here"
export TEAM_TFE_TOKEN="your_terraform_team_token_here"

# Reload
source ~/.bashrc  # or source ~/.zshrc
